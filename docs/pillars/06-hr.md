# Pillar 6 — Human Resource (HR)

> Keep employee data safe, manage leave, and (with add-ons) run shifts and self-service.

## 1. Goal & User

**Primary user:** the owner or an HR helper.
**Secondary user:** employees (via the Self-Service Mobile Leave Forms add-on).
**Job to be done:** stop storing IC photos in WhatsApp; have one place for staff records, leave balances, and shift planning.

## 2. Base Package Features

### 2.1 Core HRM Registry
A secure storage locker for employee data.

- Employee record fields:
  - Name, IC number, **IC copy** (file upload, lives in Admin Storage with sensitive flag).
  - Emergency contact profile (name, relationship, phone).
  - **Bank account routing keys** (bank name, account number) for payroll.
  - Role, employment type, start date.
- All sensitive fields encrypted at rest (see Open Questions for crypto spec).

### 2.2 Leave Overview Dashboard
Centralized leave tracking.

- Categories tracked:
  - **AL** — Annual Leave
  - **EL** — Emergency Leave
  - **MC** — Medical Certificate
- Per-employee balance + history.
- Calendar view of who's out, by day / week / month.
- Manual entry by admin in Base Package; self-service requires add-on.
- **AL Carry-Forward Rules** — configurable per business. Default policy: at year-end, unused AL carries forward up to **1.5× annual entitlement**, anything above is forfeited. Owners can change the cap (e.g. 0× = no carry-forward, 2× = generous), or set a "carry-forward must be used by Q1" expiry.

### 2.3 Malaysian Public-Holiday Calendar (State-Aware)
Malaysian public holidays vary by state — auto-populating them removes a constant source of pain in AL accruals and bookings.

- Each business sets a primary state (KUL, SGR, JHR, PNG, KDH, KTN, TRG, PRK, NSN, MLK, PHG, PLS, SBH, SWK, LBN). Holidays for that state are auto-populated for the current year and refreshed annually from the official Malaysian government list.
- Federal-only days are always included; state-specific days follow the chosen state.
- Surfaces in two places:
  - **HR Leave dashboard** — public holidays do not consume AL balance; if an employee files AL on a public holiday, the system warns.
  - **Operations Booking calendar** — public holidays appear as visual markers; owner can choose to auto-block bookings on those days (per-resource setting).
- Owners can add custom holidays (e.g. business closure for Hari Raya extension, owner's umrah trip).

### 2.4 Onboarding Checklist Template
A reusable checklist applied per new employee record.

- Owner builds one default checklist per business — typical items: collect IC copy, collect EPF/SOCSO numbers, sign employment letter, hand over uniform, brief on shop SOP, training day 1.
- When a new employee is added (§ 5.1), the checklist instantiates against that employee's profile.
- Each item: assignee (owner / employee / specific staff), due date relative to start_date, mark-done.
- Open items surface in the Admin Notification Feed (§ Admin 2.3) until completed.

### 2.5 Contract / Employment Letter Generator
Reuses the Admin Document Template engine (§ Admin 2.4) with HR fields prefilled.

- Available templates (v1):
  - **Employment Letter** (offer of employment).
  - **Confirmation Letter** (post-probation).
  - **Termination / Resignation Acceptance Letter**.
- Tap **Generate** on an employee record → opens the template with name, IC, role, start date, salary, and bank fields prefilled. Owner fills any remaining variable fields → generates PDF + secure share link → recipient signs via the **Digital Signature on Shared Documents** flow (Admin § 2.6).
- Signed PDF is filed back into the employee's record under Admin Storage with a sensitive flag.

## 4. Data Model Sketch

```
Business
 ├── employees[]
 │    ├── id (= staff_id, used in self-service URL)
 │    ├── name, ic_number (encrypted), ic_copy_file_id
 │    ├── emergency_contact { name, relationship, phone }
 │    ├── bank { bank_name, account_number (encrypted) }
 │    ├── role, employment_type, start_date
 │    └── leave_balance { al_days, el_days, mc_days_used }
 ├── leave_records[]
 │    ├── id, employee_id
 │    ├── type: AL|EL|MC
 │    ├── starts_on, ends_on, days
 │    ├── reason_text, mc_file_id (nullable)
 │    ├── status: PENDING|APPROVED|REJECTED
 │    ├── submitted_via: ADMIN|SELF_SERVICE        ← add-on
 │    └── decision_emailed_at (nullable)           ← add-on
 ├── shifts[]                                       (add-on)
 │    ├── id, employee_id, starts_at, ends_at, role
 │    └── notes
 └── self_service_form_links[]                      (add-on)
      ├── id, share_hash
      └── generated_url: bantuniaga.com/[idcompany]/leave-[secure-random-hash]
```

## 5. Key User Flows

### 5.1 Onboard an employee
1. HR → Employees → **+ New**.
2. Fill name, IC, upload IC copy, fill emergency + bank info.
3. Set initial AL / EL balances.
4. Save → employee gets a **Staff ID**.

### 5.2 Admin records a leave manually (Base)
1. HR → Leave → **+ New Record** for an employee.
2. Pick type (AL/EL/MC), dates, reason.
3. Save → balance decremented, calendar updated.

### 5.3 Self-Service leave request (add-on) — full flow
1. Owner enables the **Self-Service Mobile Leave Forms** add-on.
2. System generates a secure URL:
   ```
   bantuniaga.com/[idcompany]/leave-[secure-random-hash]
   ```
3. Owner shares this link with all staff (one link per business; staff identify themselves via **Staff ID** on the form).
4. Staff member opens the link on phone:
   - Enters Staff ID → form unlocks.
   - Picks AL / EL / MC, dates, reason.
   - For MC: attaches MC photo (uploaded to Admin Storage, sensitive flag).
   - Submits.
5. Admin Notification Feed shows: _"New leave request from [Employee Name]."_
6. Admin opens it on dashboard → reviews → taps **Approve** or **Reject**.
7. System **fires an automated email** back to the staff member's registered email with the decision.
8. Leave balance updated automatically on Approve.

### 5.4 Shift rota (add-on)
1. HR → Shift Rota → pick week.
2. Drag employee chips onto day-slot cells.
3. Publish → employees can be notified (email or in-app, TBD).

## 6. Open Questions

- Encryption scheme for IC numbers and bank accounts (KMS, per-field envelope encryption?).
- Where is staff email stored — is it required, or only when Self-Service add-on is on?
- Do staff get login accounts in v1, or only the Self-Service link with Staff ID? **Currently: link + Staff ID, no full account.**
- Payroll output (PCB, EPF, SOCSO, EIS) — explicitly out of scope at v1; planned as a future **Statutory Payroll** add-on.
- Source of truth for the public-holiday list — official gov endpoint, scraped feed, or maintained-by-us JSON?
- Onboarding checklist: do we ship a built-in default template, or does each business start with an empty checklist?
